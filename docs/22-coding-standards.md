# 22 - Coding Standards

> Product: Hublio
> Version: 1.0
> Status: Architecture Freeze v1

---

# 1. Purpose

This document defines the coding standards for Hublio.

Its goal is to ensure that all contributors write code in a consistent, maintainable, and predictable way.

Coding standards are mandatory for every package in the project.

---

# 2. Core Principles

Every piece of code should follow these principles.

* Readability over Cleverness
* Explicit over Implicit
* Simplicity over Abstraction
* Composition over Inheritance
* Small Packages
* Small Interfaces
* Fail Fast
* Dependency Injection

Code is written for humans first.

---

# 3. Project Philosophy

Hublio is a modular monolith.

The codebase follows

* Domain Driven Design
* Clean Architecture
* Package by Bounded Context

Technology must never leak into the Domain.

---

# 4. Package Rules

Each package has one responsibility.

Packages should remain cohesive.

Avoid utility packages containing unrelated functions.

Allowed

```text id="hfpm87"
internal/orchestration/domain
```

Avoid

```text id="w4cfva"
internal/utils
internal/helpers
internal/common
```

---

# 5. File Size

Recommended limits

* File ≤ 300 lines
* Function ≤ 50 lines
* Method ≤ 50 lines

Split large files before they become difficult to understand.

---

# 6. Function Design

Functions should

* Perform one task
* Return early
* Avoid deep nesting
* Avoid side effects

Prefer

```go
if err != nil {
    return err
}
```

Avoid

```go
if err == nil {
    ...
}
```

---

# 7. Interfaces

Define interfaces where they are consumed.

Keep interfaces small.

Good example

```go
type ExecutionRepository interface {
    Save(...)
    Find(...)
}
```

Avoid

```go
type Repository interface {
    Save()
    Find()
    Delete()
    Update()
    Count()
    ...
}
```

---

# 8. Dependency Injection

Construct dependencies explicitly.

Prefer constructor injection.

Example

```go
NewExecutionService(repo, publisher)
```

Avoid

* Global variables
* Service Locator
* Singleton business services

---

# 9. Error Handling

Errors are values.

Every error should contain business meaning.

Wrap infrastructure errors.

Prefer

```go
ErrConnectionDisabled
```

Avoid

```go
errors.New("failed")
```

Infrastructure errors must not leak into the API.

---

# 10. Context Usage

Every request receives a context.Context.

Use Context only for

* Cancellation
* Timeout
* Deadlines
* Request-scoped values

Do not store business data inside Context.

---

# 11. Transactions

Transactions belong to the Application layer.

Repositories should not start transactions.

One Use Case owns one transaction boundary.

---

# 12. Logging

Logging is for operators.

Log

* Errors
* Warnings
* Important state changes

Do not log

* Passwords
* API Keys
* Tokens
* Secrets

Use structured logging.

---

# 13. Events

Events describe facts.

Publish events only after successful state changes.

Events are immutable.

Never publish commands as events.

---

# 14. Connectors

Every Connector implements the same runtime contract.

A Connector owns

* Authentication
* HTTP Client
* DTO Mapping
* Error Translation

A Connector must never

* Start transactions
* Retry requests
* Publish business events
* Access the database directly

---

# 15. DTO Mapping

DTOs belong to the Interface layer.

Domain objects never expose JSON tags.

Mappings should be explicit.

---

# 16. JSON Usage

Use JSON only at system boundaries.

Examples

* REST
* Webhook
* Provider API

The Domain should not manipulate raw JSON.

---

# 17. Constants

Use typed constants whenever possible.

Avoid magic strings.

Prefer

```go
const ExecutionSucceeded = "execution.succeeded"
```

Avoid repeated string literals.

---

# 18. Time

Always store timestamps in UTC.

Use time.Time.

Never store local time.

---

# 19. UUID

All entities use UUID v7.

UUIDs are generated by the application layer.

Do not generate IDs inside repositories.

---

# 20. Concurrency

Use goroutines only when necessary.

Concurrency must be explicit.

Avoid shared mutable state.

Protect shared resources using synchronization primitives where appropriate.

---

# 21. Testing

Write tests close to the code.

Prefer table-driven tests.

Business logic should not require external services.

Mock only external dependencies.

---

# 22. Code Review Checklist

Every pull request should verify

* Correct package placement
* Dependency direction
* Business rule correctness
* Error handling
* Tests
* Logging
* No duplicated logic

Architecture violations should block the review.

---

# 23. Tooling

Required tools

* gofmt
* goimports
* golangci-lint
* govulncheck

Formatting and linting are mandatory before every commit.

---

# 24. Documentation

Exported packages should include package documentation.

Exported types and functions should include comments when they form part of the public API.

Complex business rules should be documented in code.

---

# 25. Version 1 Constraints

Version 1 intentionally excludes

* Reflection-based frameworks
* Global service containers
* ORM-generated business logic
* Hidden dependency injection
* Runtime code generation

The codebase should remain explicit and easy to debug.

---

# 26. Guiding Principles

Write code that is easy to read.

Prefer explicit design over clever abstractions.

Protect the Domain.

Keep packages small.

Keep interfaces focused.

Optimize for maintainability over brevity.
