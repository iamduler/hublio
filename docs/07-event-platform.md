# 07 - Event Platform

> Product: Hublio
> Version: 1.0
> Status: Architecture Freeze v1

---

# 1. Purpose

The Event Platform is responsible for distributing events across Hublio.

It enables platform components to react to changes without creating direct dependencies.

The Event Platform is an internal platform component.

It is not a public messaging system.

---

# 2. Responsibilities

The Event Platform owns

* Event Publishing
* Event Subscription
* Event Routing
* Event Persistence
* Event Delivery

The Event Platform does NOT own

* Business Rules
* Execution Lifecycle
* Work Queues / Worker Scheduling
* Provider Communication

Work queues belong to Platform Infrastructure.

Example

Redis Queue → Worker

The Event Platform publishes and delivers events.

It does not own the execution work queue used by Orchestration Workers.

---

# 3. Design Principles

The Event Platform follows these principles.

* Event Driven
* Immutable Events
* Loose Coupling
* Reliable Delivery
* Observable
* Connector Agnostic

Every published event represents something that has already happened.

Events never represent commands.

---

# 4. Event Categories

Version 1 defines three event categories.

## Runtime Events

Generated by the Orchestration Engine.

Examples

* ExecutionCreated
* ExecutionQueued
* ExecutionStarted
* StepStarted
* StepCompleted
* ExecutionSucceeded
* ExecutionFailed
* ExecutionCancelled

Runtime Events describe execution progress.

---

## Business Events

Represent completed business operations.

Examples

* InvoiceCreated
* InvoiceCancelled
* CustomerSynced
* ProductUpdated
* OrderImported
* PaymentReceived

Business Events describe business outcomes.

---

## System Events

Describe platform-level changes.

Examples

* ConnectorInstalled
* ConnectionActivated
* ConnectionDisabled
* CredentialRotated
* ApiKeyCreated
* UserInvited

System Events describe platform operations.

---

# 5. Event Lifecycle

Every event follows the same lifecycle.

Created

↓

Persisted

↓

Published

↓

Delivered

↓

Completed

Event history is immutable.

---

# 6. Event Structure

Every event contains

* Event ID
* Event Type
* Category
* Timestamp
* Organization ID
* Workspace ID
* Execution ID (optional)
* Correlation ID
* Payload
* Metadata

The event structure is consistent across the platform.

---

# 7. Event Publishing

Platform components publish events after completing their work.

Examples

Intent Processor

↓

IntentAccepted

Orchestration Engine

↓

ExecutionStarted

Connector Runtime

↓

InvoiceCreated

Publishing should occur after state changes are committed.

---

# 8. Event Subscription

Platform components subscribe to relevant events.

Examples

Observability

* Runtime Events

Audit

* Runtime Events
* System Events

Notification

* Business Events

Subscribers remain independent of publishers.

---

# 9. Event Flow

```text
Business Intent
        │
        ▼
Orchestration Engine
        │
        ▼
Runtime Event
        │
        ▼
Event Platform
        │
   ┌────┼────┬──────────┐
   ▼    ▼    ▼          ▼
Audit Metrics Logs Notifications
```

The Event Platform distributes events.

It never performs business processing.

---

# 10. Event Persistence

Every published event is persisted.

Persisted events support

* Audit
* Troubleshooting
* Replay (internal)
* Historical Analysis

Events are append-only.

---

# 11. Event Ordering

Events belonging to the same Execution should preserve publication order.

Ordering across unrelated Executions is not guaranteed.

---

# 12. Event Delivery

Version 1 uses at-least-once delivery.

Subscribers should be idempotent.

Duplicate event processing must not create duplicate business operations.

---

# 13. Error Handling

If event delivery fails

* Retry according to platform policy
* Record delivery failure
* Emit observability metrics

Failed deliveries must never interrupt the originating Execution.

---

# 14. Observability

Every event exposes

* Event ID
* Event Type
* Category
* Execution ID
* Correlation ID
* Timestamp
* Delivery Status

Every event should be traceable.

---

# 15. Security

Events must never contain

* Passwords
* API Keys
* Access Tokens
* Secrets
* Private Keys

Sensitive business fields should be masked when appropriate.

---

# 16. Platform Boundaries

The Event Platform owns

* Event Distribution
* Event Persistence
* Event Delivery

The Event Platform does not own

* Business Decisions
* Provider Communication
* Execution State
* Business Validation
* Work Queues / Worker Infrastructure

Responsibilities remain clearly separated.

---

# 17. Version 1 Constraints

Version 1 intentionally excludes

* Event Streaming Platform
* Kafka Integration
* Event Sourcing
* Cross-region Event Replication
* Public Event Bus

These capabilities may be introduced in future versions without changing the Event Platform contract.

---

# 18. Guiding Principles

Events describe facts.

Events are immutable.

Events are append-only.

Publishers never know subscribers.

Subscribers never depend on publishers.

The Event Platform exists to synchronize platform components while keeping them loosely coupled.
